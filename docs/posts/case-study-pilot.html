Pilot: a self-hosted app store built with Mash

Pilot is the reference application for the Mash SDK. The pool is the catalog, agents are the listings, host compositions are the installed apps, and a terminal CLI is the storefront. Here's how it's built.

What Pilot does

Pilot ships a catalog of agents the user can browse, compose into teams, and enter from a terminal. Its featured application is a multi-agent guide to the Mash codebase: one primary agent plus six module-scoped copilots, each loaded with the documentation and source for its slice of the repo. The user composes which agents work together, then enters the REPL.

It also serves as the application behind the Mash documentation series, so every framework feature is observable in a real running system: delegation, host composition at runtime, durable workflows, skills, human-in-the-loop approval, MCP, and traces.

The catalog

Every agent the store ships is a CatalogEntry in pilot/catalog/__init__.py. Each entry pairs two callables: a spec factory that constructs an AgentSpec (accepting workspace_root) and a metadata factory that returns the AgentMetadata for that agent's listing.

# pilot/catalog/__init__.py
from dataclasses import dataclass

@dataclass(frozen=True)
class CatalogEntry:
    agent_id: str
    create_spec: Callable[..., AgentSpec]   # accepts workspace_root=...
    build_metadata: Callable[[], AgentMetadata]

CATALOG: tuple[CatalogEntry, ...] = (
    CatalogEntry("pilot",            pilot.create_spec,             pilot.build_metadata),
    CatalogEntry("cli-copilot",      cli.create_cli_copilot_spec,   cli.build_cli_metadata),
    CatalogEntry("api-copilot",      api.create_api_copilot_spec,   api.build_api_metadata),
    CatalogEntry("mcp-copilot",      mcp.create_mcp_copilot_spec,   mcp.build_mcp_metadata),
    CatalogEntry("runtime-copilot",  runtime.create_runtime_copilot_spec, runtime.build_runtime_metadata),
    CatalogEntry("workflow-copilot", workflow.create_workflow_copilot_spec, workflow.build_workflow_metadata),
    CatalogEntry("admin-copilot",    admin.create_admin_copilot_spec, admin.build_admin_metadata),
)

build_pool() in pilot/spec.py iterates CATALOG, calls HostBuilder.agent() for each entry, defines the default guide host, and registers the quiz workflow on the built pool. The builder includes Masher and its workflows automatically.

# pilot/spec.py
def build_pool(workspace_root: Path | None = None) -> AgentPool:
    builder = HostBuilder()
    for entry in CATALOG:
        builder.agent(
            entry.create_spec(workspace_root=ws), metadata=entry.build_metadata()
        )
    builder.host(Host(host_id="guide", primary="pilot", subagents=(...)))
    pool = builder.build()
    pool.register_workflow(build_quiz_workflow_spec(resolved))
    return pool

The guide composition ships in code as the default. Further hosts belong to the CLI's config file (~/.pilot/hosts.json) and are published to the deployment as idempotent PUTs when the CLI connects. Which agents work together is user configuration.

The guide family

The seven catalog agents form the guide family. The pilot agent is the primary: it covers the shared and cross-cutting modules (core, tools, skills, logging, memory) and delegates to module-scoped copilots for everything else.

Agent Scope
pilotShared/cross-cutting: core, tools, skills, logging, memory
cli-copilotsrc/mash/cli — commands, REPL, terminal rendering
api-copilotsrc/mash/api — HTTP routes, FastAPI, admin dashboard
mcp-copilotsrc/mash/mcp — MCP client/server, transport, tool adaptation
runtime-copilotsrc/mash/runtime — request lifecycle, event sourcing, durability
workflow-copilotsrc/mash/workflows — step pipelines, DBOS orchestration, run status
admin-copilotsrc/mash/api/routes/admin — admin dashboard routes and assets

Each copilot is a subclass of CopilotAgentSpec. The shared base loads the relevant README.md and AGENTS.md files from its doc_roots into the system prompt with Anthropic prompt caching. Copilots run with conversation_history_turns=0 and temperature=0.2 — they receive a single delegated question, answer it with bash if needed, and return.

# pilot/catalog/_base.py — shared copilot config
def _build_copilot_config(self, agent_id: str) -> AgentConfig:
    return AgentConfig(
        app_id=agent_id,
        system_prompt=self.build_system_prompt(),  # prompt-cached doc blocks
        skills_enabled=False,
        conversation_history_turns=0,
        max_steps=50,
        temperature=0.2,
    )

The AgentMetadata each copilot registers lists its capabilities and usage_guidance. The primary reads that metadata when routing a delegated question — the same metadata that appears in pilot browse listings. The storefront and the delegation directory are the same data.

The pilot agent

The pilot primary carries more tools than the copilots. It registers a BashTool scoped to the workspace, an UpdateDocsTool that requires user approval, and an AskUserTool. It also optionally connects to a GitHub MCP server when GITHUB_MCP_PAT is set, with a two-tool allowlist (list_commits and get_commit) for inspecting mashpy commits.

pilot (primary)
Tools: bash · update_docs · ask_user · GitHub MCP (optional)
Skills: build-mash-agent · build-mash-workflow · build-mash-host
InvokeSubagent
Copilots (subagents)
cli · api · mcp · runtime · workflow · admin
Tools: bash (read-only workspace access)

The update_docs tool has requires_approval = True, which causes the runtime to pause and surface an approval event before the tool runs. In the REPL this appears as an interactive prompt; over the API it emits a request.waiting event that the client polls. The user approves or rejects; the runtime resumes or aborts. This is the human-in-the-loop flow exercised live.

Workflows

Pilot ships one workflow. pilot-quiz is a Mash step pipeline: build_quiz_workflow_spec() returns a WorkflowSpec whose single AgentStep carries QuizAgentSpec as its agent_spec, so registering the workflow also registers the quiz agent as a workflow-only runtime, hidden from listings and delegation.

# pilot/catalog/workflows/quiz.py
def build_quiz_workflow_spec(workspace_root: Path | None = None) -> WorkflowSpec:
    return WorkflowSpec(
        workflow_id="pilot-quiz",
        steps=[
            AgentStep(
                step_id="run-quiz",
                agent_spec=QuizAgentSpec(resolved),
                output={"type": "object"},
            )
        ],
    )

A host config that includes "workflows": ["pilot-quiz"] makes the /quiz REPL command available to anyone who enters that host. The command starts a run over the workflow API and streams it: the quiz agent loads its mash-quiz skill, asks its questions through durable AskUser interactions, and the REPL renders each one as an interactive prompt. The run survives a host restart mid-question the same way any durable request does.

Deployment

Pilot ships as a single Docker container with an embedded Postgres database. The container runs mash host serve --host-app pilot.spec:build_pool. One volume keeps the database, agent ledger, and session history across restarts.

docker run -d --name pilot -p 8000:8000 \
  -e ANTHROPIC_API_KEY=sk-ant-... \
  -v pilot-data:/var/lib/pilot \
  ghcr.io/imsid/mash-pilot:latest

Add -e GITHUB_MCP_PAT=ghp_... to enable the GitHub commit tools on the pilot agent. Set MASH_DATABASE_URL to bring an external Postgres instead of the embedded one — this is the only change the Docker Compose development setup makes. The CLI installs separately and points at any running store by URL.

curl -fsSL https://raw.githubusercontent.com/imsid/mash-pilot/main/install.sh | sh
pilot browse                  # see the catalog
pilot repl --host guide       # enter the default composition

The admin dashboard is available at /admin on the running host. It shows the pool, active host compositions, and agent traces with a span tree drawer.

Adding an agent to the catalog

A new catalog entry is a new package under pilot/catalog/agents/ plus one CatalogEntry in CATALOG. Everything else — pool registration, store listing, delegation routing, session management, streaming — comes from the host.

1
Create pilot/catalog/agents/my_agent/spec.py. Subclass AgentSpec (or CopilotAgentSpec for a module-scoped guide) and implement get_agent_id(), build_tools(), build_llm(), and build_agent_config(). Add a build_metadata() function returning AgentMetadata.
2
Add a CatalogEntry to CATALOG in pilot/catalog/__init__.py: one tuple of (agent_id, create_spec, build_metadata).
3
Merge to main and redeploy the container. The agent appears in pilot browse immediately.
4
Compose it into a host: pilot compose my-host --primary pilot --subagents my-agent. Enter: pilot repl --host my-host.
The stock mash CLI speaks the same protocol as pilot. mash connect --api-base-url http://127.0.0.1:8000 --agent cli-copilot opens a direct REPL against any catalog agent, bypassing the store CLI entirely — useful for debugging a new agent before composing it into a host.
