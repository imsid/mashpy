Moma: a Node.js app backed by a Mash agent host

Moma is an AI companion app for older adults. Its frontend and application logic are a Node.js service on Vercel; its AI workloads run in a separate Python service built with Mash. Here's how the two fit together.

What Moma does

Moma surfaces two AI-driven features. A Deep Dive generates a personalized article each day on a topic drawn from the user's interest profile — researched, written, and structured by an agent. A Chat lets the user have a conversation about anything at their own pace, with a warm and patient tone tuned for someone new to AI.

Both features required the same underlying infrastructure: a place for session history to live, a way to get structured output back from a model, a streaming event model so the browser doesn't hang waiting for a long LLM call. The question was where that infrastructure lived and who owned it.

Separating the app layer from the agent layer

Moma's Node.js codebase handles the things a web app is already good at: auth, database reads and writes, topic selection, media enrichment (Wikipedia images, YouTube videos). It does not call any LLM directly. All AI work goes to a separate Python service — the Mash host — over HTTP.

The two services deploy independently and scale independently. Moma auto-deploys on every push to main. The Mash host has its own GitHub Actions pipeline that triggers only when files under agents/ change — building a Docker image, pushing to GCR, and deploying to Cloud Run. A UI change doesn't touch the agent container; a new agent doesn't redeploy Vercel.

One integration point

All of Moma's communication with the Mash host goes through a single file: lib/mash.js. It implements two functions — askAgent() and runWorkflow() — each following the same submit-then-poll-SSE pattern.

The H2A protocol underneath both is plain HTTP and Server-Sent Events. Submit a request, get back a request_id or run_id, then stream events until request.completed. Any HTTP client in any language can consume a Mash host. The mash connect CLI speaks the same protocol — so during development, the same chat agent Moma calls in production is reachable directly from a terminal.

Chat: agent request

api/chat.js calls askAgent(). It posts the user's latest message to the host with a session ID, and receives back a text response.

// 1. Submit
POST /api/v1/agent/chat/request
{ "message": "What is AI?", "session_id": "moma-chat-alice-2026-06-28" }
→ 202  { data: { request_id: "r_abc" } }

// 2. Stream
GET  /api/v1/agent/chat/request/r_abc/events
→ event: agent.trace
→ event: request.completed
  data: { response: { text: "Here's a simple explanation…" } }

// 3. Return to browser
{ reply: "Here's a simple explanation…" }

The session ID is the only thing Moma needs to maintain across turns. It sends one message at a time — no full messages array, no history management. The Mash host holds the conversation history server-side, keyed to that session ID.

Deep Dive: workflow run

api/deepdive.js calls runWorkflow(). It submits structured inputs — a topic seed, recent article titles, and web grounding from a prior search — and receives back a fully typed, schema-validated article object.

// 1. Submit inputs
POST /api/v1/workflow/deepdive/run
{ "input": { "seed": "The Mughal Empire", "recentTitles": [...], "grounding": "…" } }
→ 202  { data: { run_id: "run_xyz" } }

// 2. Stream
GET  /api/v1/workflow/deepdive/runs/run_xyz/events
→ event: request.completed
  data: { response: { structured_output: { title: "…", sections: [...] } } }

// 3. Enrich and persist
→ fetch Wikipedia image + YouTube video
→ INSERT deep_dives, deep_dive_sections, user_deep_dives

The Deep Dive agent runs with claude-sonnet-4-6 and a deepdive skill — a named bundle of web search and article-authoring instructions. Its system prompt is one line: Run the deepdive skill. The skill's own instructions drive the behavior. The Pydantic DeepDiveOutput model is the single source of truth for the response schema; Mash enforces it at runtime, and api/deepdive.js receives the validated object without constructing a prompt or defining a schema on the JS side.

The Mash host: two agents, one pool

The host registers a ChatAgent and a deepdive workflow backed by DeepDiveAgent. Both are wired in agents/host/spec.py:

class ChatAgent(AgentSpec):
    def get_agent_id(self):       return "chat"
    def build_llm(self):          return AnthropicProvider(app_id="chat", model="claude-haiku-4-5-20251001")
    def build_agent_config(self): return AgentConfig(app_id="chat", system_prompt=SYSTEM_PROMPT, max_tokens=500)
    def build_tools(self):        return ToolRegistry()
    def build_skills(self):       return SkillRegistry()

def build_pool():
    builder = HostBuilder()
    builder.agent(ChatAgent(), metadata=CHAT_METADATA)
    builder.workflow(WorkflowSpec(
        workflow_id="deepdive",
        tasks=[TaskSpec(task_id="generate", agent_spec=DeepDiveAgent(),
               structured_output=DEEPDIVE_STRUCTURED_OUTPUT)],
        task_message=WorkflowTaskMessageSpec(skill_name="deepdive"),
    ))
    return builder.build()

run_host(build_pool())  # starts FastAPI server

The host runs in a Docker container on Cloud Run, started with mash host serve. Mash auto-migrates its own schema on startup — session history, durable request events, and agent traces all live in a moma_mash database on shared Cloud SQL. Moma's own database (moma) sits on the same instance and is managed separately.

Deployment and scaling

Moma deploys on Vercel as a serverless Node.js application. Every push to main triggers an automatic redeploy; feature branches get Vercel preview URLs automatically. Vercel scales at the function level and is completely unaware of the Mash host.

The Mash host is a Dockerized FastAPI service on Google Cloud Run, started with mash host serve:

# agents/Dockerfile
CMD ["sh", "-c", "mash host serve \
  --host-app host.spec:build_pool \
  --host ${MASH_API_HOST} \
  --port ${MASH_API_PORT} \
  ${MASH_API_KEY:+--api-key \"$MASH_API_KEY\"}"]

Cloud Run keeps one container warm at all times (--min-instances 1) to avoid cold-start latency on the first request of the day. It scales on request volume to the Mash host, independently of how many Vercel function invocations are happening.

The two services have separate CI/CD pipelines. Moma's pipeline fires on any push. The Mash host pipeline triggers only when files under agents/ change:

# .github/workflows/deploy-agents.yml
on:
  push:
    branches: [main]
    paths: [agents/**]   # only fires when agents/ changes

Both databases live on the same Cloud SQL instance. Moma manages its own schema via migration files. The Mash host runs its own schema migrations on startup — session history, durable request events, and agent traces are in a separate moma_mash database that Mash owns entirely.

Adding a new agent

Because the host owns the agent runtime, adding a capability is entirely self-contained in the agents/ directory. Moma's Node code doesn't change.

1
Create agents/host/catalog/my_agent.py — subclass AgentSpec, declare the model, system prompt, tools, and skills.
2
Register it in agents/host/spec.py: one line, builder.agent(MyAgent(), metadata=METADATA).
3
Merge to main. GitHub Actions rebuilds and redeploys the Cloud Run container.
4
Call it from any Moma route: await askAgent('my_agent', message, sessionId).

Session history, durability, streaming, and structured output come from the host. The new agent file contains only what's specific to that agent's behavior.

The mash connect CLI speaks the same H2A protocol as Moma's Node backend. During development, mash connect --api-base-url http://127.0.0.1:8000 --agent chat opens an interactive REPL against the local container — the same session management, streaming, and history that Moma's production users see.
